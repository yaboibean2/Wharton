SINGLE STOCK ANALYSIS - COMPREHENSIVE OVERVIEW

OVERVIEW

The main single-stock analysis is driven by the app UI and executed by the PortfolioOrchestrator. The app collects the ticker, date, and weighting mode, stores those inputs in session state, then reruns into the analysis path. The real engine logic lives in PortfolioOrchestrator.analyze_single_stock().

At a high level, the flow is:

1. Initialize the orchestrator with configs and agent objects
2. Collect analysis inputs from the UI
3. Pass per-run weights and theory settings into the orchestrator
4. Gather stock data in parallel
5. Run five analysis agents in parallel
6. Blend the five agent scores according to the selected weighting mode
7. Produce a final score and recommendation
8. Return a result payload for display

3. MacroRegimeAgent

These are created in the constructor at:

The orchestrator:
- uses a thread pool with 5 workers
- submits all 5 analyze() calls
- lightly staggers launches by 0.2 seconds to avoid API bursts
- records each agent’s elapsed time
- updates progress as each agent completes

If an agent succeeds:
- its result is stored in agent_results
- missing score defaults to 50
- a completion message is generated

If an agent fails:
- the orchestrator inserts a fallback neutral result with score 50

That fallback behavior prevents one broken agent from collapsing the entire analysis.

THE CORE BLENDING FORMULA

The weighted-average blend inside _blend_scores() is:

base_score = sum(score * weight) / sum(weight)

More formally:

base score = weighted average of agent scores using the effective weights for that run

If the total weight is zero, the method falls back to 50.

MODE 1 - EQUAL WEIGHTS

The equal-weight preset is defined in:
app.py lines 2232-2245

The app passes:
- value = 1.0
- growth_momentum = 1.0
- macro_regime = 1.0
- risk = 1.0
- sentiment = 1.0

Inside the orchestrator, those are remapped to the agent-key names and temporarily applied.

What equal mode means:
- each of the 5 agents has identical influence in the weighted average
- so the base weighted score becomes the simple mean of the five scores

However, equal mode is NOT a pure average all the way to the end.

After the equal-weight base score is computed, the engine still enters the non-theory upside multiplier path in:
engine/portfolio_orchestrator.py lines 581-648

That means the final score is:

final score = equal-weight average, then adjusted by the upside multiplier

UPSIDE MULTIPLIER IN NON-THEORY MODES

In equal mode and custom mode, _blend_scores() adds a second layer after the weighted average.

This layer examines:

1. Growth/Momentum score
- >= 80 -> +15%
- >= 70 -> +10%
- >= 60 -> +5%
- < 40 -> -5%

2. Sentiment score
- >= 75 -> +8%
- >= 65 -> +5%

3. Value score
- >= 75 -> +5%

4. Risk score
- < 30 -> -10%

Then the total multiplier is clamped:
- max 1.35
- min 0.85

So in equal mode:
- all agents count equally in the base score
- then growth, sentiment, value, and extreme risk can further modify the final output

Interpretation:
Equal mode is equal at the averaging stage, but not fully neutral at the final-score stage.

MODE 2 - CUSTOM WEIGHTS

The custom-weight preset lives in:
app.py lines 1990-2124

UI behavior:
- initializes all sliders at 0.5
- sliders range from 0.0 to 1.0
- step size is 0.05

The custom weights are stored in session state and passed as:
- value
- growth_momentum
- macro_regime
- risk
- sentiment

The orchestrator remaps those keys and applies them temporarily.

Important detail:
The engine does not pre-normalize custom weights before blending. It uses them directly in the weighted average formula.

That means only the relative proportions matter.

Examples:
- all five weights at 0.5 is mathematically equivalent to all five at 1.0
- setting growth_momentum to 1.0 and all others to 0.25 gives growth four times as much influence as each other factor
- setting a factor to 0.0 removes that factor from the base weighted average

Then, just like equal mode, custom mode still goes through the non-theory upside multiplier.

So custom mode is a two-layer process:
1. user-defined weighted average
2. engine-defined upside/risk adjustment

Final interpretation:
Custom mode lets the user choose the relative importance of each agent, but the final score is still stylized by the engine’s upside philosophy.

HOW THEORY MODE WORKS INSIDE THE ENGINE

Inside _blend_scores():

If regime_modulation=True:
1. the engine reads the detected regime from the macro agent result
2. it calls _apply_regime_modulation()
3. it adjusts the base weights according to the regime table
4. it renormalizes them
5. it computes the weighted average

WEIGHT NORMALIZATION IN THEORY MODE

_apply_regime_modulation() is implemented in:
engine/portfolio_orchestrator.py lines 690-719

It does the following:
- applies regime shifts to the base weights
- floors each weight at about 0.02
- renormalizes all weights so they sum to 1.0
- repeats clamp/normalize a few times if necessary

This makes theory mode more mathematically self-contained than the other modes.

PRACTICAL DIFFERENCE BETWEEN THE THREE MODES

EQUAL WEIGHTS
- base score: simple mean of the 5 agent scores
- final score: equal-weight average plus upside/risk multiplier
- philosophy: neutral averaging, but still with engine upside tilt

CUSTOM WEIGHTS
- base score: user-controlled weighted average
- final score: custom weighted average plus upside/risk multiplier
- philosophy: user-selected emphasis, still with engine upside tilt

THEORY BASED
- base score: theory weight table, shifted by macro regime
- final score: pure regime-adjusted weighted average
- philosophy: factor-model-driven scoring with no extra upside overlay

WHY THEORY MODE FEELS DIFFERENT

Theory mode differs from equal/custom in one very important way:
- it disables the upside multiplier

So if two stocks have the same agent scores:
- equal/custom may still produce a boosted or penalized final score because of the overlay
- theory mode will not

That makes theory mode cleaner and more interpretable for anyone who wants the final score to reflect only the weighted agent mix.

IMPORTANT CAVEATS

1. DEFAULT CONFIG MISMATCH
The app loads both model config and IPS config, but the orchestrator’s live default weights come from IPS config, not model config.

Relevant files:
utils/config_loader.py lines 18-35
config/model.yaml lines 1-6
config/ips.yaml lines 1-120
engine/portfolio_orchestrator.py lines 147-163

Since ips.yaml currently has no agent_weights, the live runtime defaults are the hardcoded 20/40/10/15/15 split.

2. EQUAL MODE IS NOT FULLY NEUTRAL
Equal mode only equalizes the weighted average stage. The final score still gets the upside/risk overlay.

3. CUSTOM MODE IS NOT PURELY CUSTOM
Custom mode changes the weighted average, but the final score is still modified by the engine’s non-theory upside multiplier.

4. THEORY MODE IS A UI PACKAGING CONCEPT
The engine only sees weights plus regime_modulation=True. The phrase “Theory Based” comes from the UI preset, not from a special engine type.

5. BLENDED SCORE TERMINOLOGY IS SLIGHTLY CONFUSING
The engine stores the output of _blend_scores() as both blended_score and final_score, but the UI breakdown may separately recompute a raw weighted average for explanation.

BOTTOM LINE

The main single-stock analysis works like this:

- collect one year of stock data and benchmark context
- run five independent specialist agents in parallel
- compute five agent scores
- combine them using one of three weight systems
- produce a final score and recommendation

The three modes differ mainly in the score-combination step:

Equal Weights
- every agent has equal base influence
- final score still uses upside/risk multiplier

Custom Weights
- user controls relative agent influence
- final score still uses upside/risk multiplier

Theory Based
- starts from a factor-theory weight table
- adjusts weights based on detected macro regime
- disables upside multiplier
- final score is a pure regime-adjusted weighted average
